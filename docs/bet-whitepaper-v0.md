## Оговорка

Документ описывает **продуктовый замысел** крипто-слоя MMO. Это **не** инвестиционная рекомендация, не проспект эмиссии и не юридическое заключение. Конкретные числа (эмиссия, комиссии, сроки) — **иллюстративные**, до фиксации с юридической стороны и аудита смарт-контрактов.

Цель v0: **единая формулировка** utility-токена и NFT, правил сжигания и очередности работ для команды и партнёров.

---

## 1. Резюме

- **BET** — утилитарный токен (не обещание прибыли): турниры, косметика и недвижимость в игре, коллективное открытие контента, часть экономики аукциона. **Нельзя** покупать за BET боевую силу или эксклюзивный доступ к базовому контенту (см. [crypto-economy.md §1.2](crypto-economy.md)).
- **Дефляция** — за счёт **burn** из комиссий, турниров, открытия зон, косметики и др. (дизайн — §1.3 в crypto-economy).
- **NFT** — визуал, статус, удобство; **боевые предметы** — soulbound, **не** торгуются. Земля, дома, скины — продаваемые коллекции (§1.4).
- **Технологии (целевой ориентир):** BET как **ERC-20**, продаваемые активы **ERC-721**, soulbound **ERC-1155** без transfer; аукцион, контракт «открытия зон», турниры, мультисиг-казначейство; **газлесс** через релей для массового игрока.
- **Форки:** открытый game/core; контракты BET/NFT — обязательный слой; роялти в казну (дизайн §1.10 crypto-economy).

## 2. Очередность внедрения (зафиксированная для v0)

Согласована с [crypto-economy.md §2 «Рекомендуемая очередность»](crypto-economy.md) и [§3 v0](crypto-economy.md):

| Фаза | Содержание |
|------|------------|
| **A** | Принятие **whitepaper v0** (этот документ) + юридический/продуктовый review параметров |
| **B** | **Смарт-контракты** (тестнет): BET ERC-20 (фикс. эмиссия, контролируемый mint игровым ядром, публичный burn) |
| **C** | **Индексатор** событий → **PostgreSQL** / кэш **Redis**, RPC для игрового бэкенда |
| **D** | **Газлесс-релей** (подпись, мониторинг) |
| **E** | Безопасность: роли, pause, anti-whale, нагрузочные тесты |
| **F** | **API / Unity:** embedded wallet, покупки, аукцион, зоны (поверх синхронизированной БД) |
| **G** | Публичные адреса контрактов, дашборд казны, обновление whitepaper до **v1** перед публичными средствами |
| **H** | Внешний **аудит** контрактов перед mainnet/листингом |

Параллельно с **B–D** возможна подготовка схем БД и BFF под ончейн-события (без публичного приёма средств).

## 3. Критерий «утверждено v0»

- Командой зафиксированы: utility vs non-security, сценарии burn, разделение NFT, роль релея, порядок фаз **A–H**.
- Следующий **обязательный** шаг инженерии после sign-off: **фаза B** (тестнет) + задел под **C** в `backend/` (отдельные задачи, вне scope этого PR-дока).
